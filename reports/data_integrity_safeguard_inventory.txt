==================================================
DEFENSIVE POSTURE ASSESSMENT
==================================================

Overall Defensive Posture: MODERATE-STRONG

The pipeline has strong per-layer protections (monotonicity, dedup,
atomic writes, validation gates, hash-based versioning). The recently
added delta guard closes the MN1 contamination vector.

However, cross-layer coordination is weak. The system does not detect
when upstream corrections invalidate downstream datasets.


BLIND SPOTS IDENTIFIED:
--------------------------------------------------
1. NO DOWNSTREAM INVALIDATION TRIGGER
   When RAW is corrected (e.g., SPX500 7->38 rows), CLEAN and RESEARCH
   remain stale. The timestamp skip logic (max_raw_ts <= max_clean_ts)
   actively PREVENTS rebuilds when CLEAN has future timestamps from
   corrupt data. Requires manual delete + rebuild.

2. NO ROW COUNT SANITY CHECK
   Neither the validator nor the ingestion path checks if row count
   changes dramatically between runs (e.g., 38 -> 7 rows). A broker
   returning vastly fewer rows would pass all current checks.

3. NO FUTURE TIMESTAMP BLOCKING IN VALIDATOR
   The validator checks monotonicity and gaps but does NOT reject
   timestamps beyond current date. Future dates from broker pass through.

4. NO CROSS-LAYER ROW COUNT COMPARISON
   RAW(38) vs CLEAN(7) mismatch is never detected. Each layer only
   checks its own internal consistency.

5. DELTA GUARD ONLY COVERS D1, 4H, 1H
   Lower timeframes (1m, 5m, 15m, 30m) have no median delta guard.
   Monthly contamination on 15m data would pass silently.

6. NO PERSISTENT AUDIT LOG
   Validation failures, guard triggers, and blocked writes are only
   printed to stdout. If terminal output is not captured, events are lost.


RECOMMENDATIONS:
--------------------------------------------------
1. Add downstream staleness detector:
   After RAW write, compare RAW row_count / last_ts against CLEAN.
   If RAW has more rows or newer last_ts -> flag CLEAN as stale.

2. Add future timestamp blocker:
   In validator or delta guard, reject any timestamp > NOW + 24h.

3. Add row count regression guard:
   If existing file has N rows and new data produces < N*0.5 rows
   for the same year, block the write.

4. Extend delta guard to all timeframes:
   Add thresholds for 1m (> 0.01 days), 5m (> 0.05 days),
   15m (> 0.1 days), 30m (> 0.15 days).

5. Add persistent audit event log:
   Write JSON events for: guard triggers, validation failures,
   blocked writes, skipped files. File: LOGS/audit_events.jsonl

6. Add cross-layer row count assertion:
   After CLEAN rebuild, assert CLEAN rows ~= RAW rows for same
   year file. Large divergence -> alert.
